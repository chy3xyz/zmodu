# ZigModu Build — zmodu First Principle

**zmodu generates everything possible. AI writes ONLY what zmodu cannot.**

## ⛔ HARD BLOCK

Files marked `//! ⛔ @generated by zmodu — DO NOT EDIT` are **read-only**. Never edit them. To change behavior:
1. Update `schema.sql`
2. Re-run `zmodu scaffold --sql schema.sql --name <name>`
3. AI only writes in `ext/`, `src/shared/`, `src/business/`, `src/compat/`

## Input → Mode Selection

```
Input is SQL file?
  → Mode 2: Brownfield
Input is live database (PostgreSQL/MySQL/SQLite)?
  → Mode 2b: Brownfield (from DB) — use --from-db
Input is existing project directory (Java/PHP/Go/Rust)?
  → Mode 3: Migration — invoke zigmodu-analyze first
Input is requirements description (text)?
  → Mode 1: Greenfield
```

## Scaffold Variants

```bash
# From SQL file (parse DDL)
zmodu scaffold --sql schema.sql --name <name> --json-style camel

# From live database (introspect via information_schema/PRAGMA)
zmodu scaffold --from-db "postgresql://user:pass@host:5432/db" --name <name>
zmodu scaffold --from-db "mysql://user:pass@host:3306/db" --name <name>
zmodu scaffold --from-db /path/to/db.sqlite --name <name>

# Import SQL → database → introspect (validates SQL, gets accurate types)
zmodu scaffold --sql schema.sql --from-db sqlite:///fresh.db --name <name>

# Subsystem detection (auto from table prefixes like shop_*, crm_*)
# Generates: src/modules/<subsystem>/<module>/ with nested imports
```

## Mode 1: Greenfield (requirements → project)

```
Pipeline:
  1. AI designs schema.sql from requirements
  2. zmodu scaffold --sql schema.sql --name <name> --json-style camel
  3. zig build (MUST 0 errors)
  4. AI writes: ext/service.zig (business rules), ext/api.zig (custom endpoints)
  5. zig build test (MUST all pass)
```

## Mode 2: Brownfield (SQL → project)

```
Pipeline:
  1. Verify SQL: grep "CREATE TABLE" schema.sql → N tables confirmed
  2. zmodu scaffold --sql schema.sql --name <name> --json-style camel
  3. zig build (MUST 0 errors)
  4. AI gap analysis per module:
     - FK relationships → cross-module queries in ext/service.zig
     - Status fields → state machine in ext/service.zig
     - Auth fields → permission checks in ext/api.zig
  5. AI writes per module: ext/service.zig + ext/api.zig
  6. zig build test
```

## Mode 3: Migration (Java/PHP/Go/Rust → ZigModu)

```
Pipeline:
  1. Invoke zigmodu-analyze → outputs: schema.sql, routes.json, module-map.json
  2. zmodu scaffold --sql schema.sql --name <name> --json-style camel
  3. zig build (MUST 0 errors)
  4. Route matching: compare generated api.zig with routes.json
     - Mismatched paths → adjust routes in generated api.zig (path only, not logic)
     - Missing routes → ext/api.zig
  5. Invoke zigmodu-translate per source file → ext/service.zig with [AUTO]/[REVIEW]/[MANUAL]
  6. compat/ layer for legacy response format compatibility
  7. zig build test
```

## Project Structure (post-scaffold)

```
src/
├── main.zig              # ⛔ entry point (don't edit)
├── shared/               # ✅ shared kernel
├── business/             # ✅ cross-module orchestration
├── compat/               # ✅ legacy compatibility
└── modules/
    ├── <name>/            # flat module (no subsystem)
    │   ├── module.zig     # ⛔ lifecycle + barrel + health
    │   ├── model.zig      # ⛔ struct(sql_table) + json_names
    │   ├── persistence.zig # ⛔ ORM Repository(T)
    │   ├── service.zig    # ⛔ CRUD + validate
    │   ├── api.zig        # ⛔ REST {id} handlers
    │   └── ext/           # ✅ AI writes here
    └── <subsystem>/       # auto-detected (tables share prefix)
        └── <module>/      # same 5 files + ext/ structure
```

### Subsystem Detection

Tables with shared first-segment prefix auto-grouped:
- `shop_orders`, `shop_products` → subsystem `shop`, modules `orders` + `products`
- `crm_customers`, `crm_leads` → subsystem `crm`, modules `customers` + `leads`
- `users` (no shared prefix) → flat module `users`

Imports for subsystem modules:
```zig
const shop_orders = @import("modules/shop/orders/module.zig");
const crm_customers = @import("modules/crm/customers/module.zig");
```

## AI Edit Rules

### ✅ Safe (survives regeneration)
```
src/modules/<name>/ext/service.zig
src/modules/<name>/ext/api.zig
src/shared/*.zig
src/business/*.zig
src/compat/*.zig
tests/
```

### ⛔ NEVER EDIT (overwritten on regeneration)
```
Files starting with "//! ⛔ @generated by zmodu"
→ model.zig, persistence.zig, service.zig, api.zig, module.zig, main.zig, build.zig, build.zig.zon
```

## Verify After Every Change

```bash
zig build          # 0 errors or STOP
zig build test     # all pass or FIX
```

## Blocking Rules

```
BLOCKED if:
  - schema.sql empty after analysis → need DB credentials or migration files
  - zig build has errors → fix compilation before proceeding
  - zmodu not installed → "cd zmodu && zig build"
  - AI attempts to edit @generated file → STOP, redirect to ext/ or schema.sql
```
